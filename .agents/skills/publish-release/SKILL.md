## When to Use

Load this skill when:
- User asks to release a new version, bump version, or publish to GitHub/Homebrew
- User says "subir versión", "nueva versión", "nuevo release", "publicar versión", or "crear release"
- User asks to tag a release or increment the version number

---

## Semantic Versioning Rules

This project follows [SemVer](https://semver.org) strictly:

| Change type | Version segment | Examples |
|-------------|----------------|----------|
| Breaking change to MCP tool API, protocol, or config schema | **MAJOR** (X) | `1.x.x → 2.0.0` |
| New MCP tool, new CLI command, new dpf job type (backwards-compatible) | **MINOR** (Y) | `1.0.x → 1.1.0` |
| Bug fix, refactor, docs update, dependency update, CI fix | **PATCH** (Z) | `1.0.1 → 1.0.2` |

Determine the correct segment **before** making any changes.

## Files That Must Be Updated for Every Release

| File | What to change | Example |
|------|---------------|---------|
| `VERSION` | Plain semver string (no `v` prefix) | `1.0.2` |
| `README.md` | Badge URL: `version-X.Y.Z-blue.svg` | `version-1.0.2-blue` |
| `cmd/devforge-mcp/main.go` | `mcpserver.NewMCPServer("devforge", "X.Y.Z", ...)` | `"1.0.2"` |
| `Formula/devforge.rb` | `version "X.Y.Z"` | `version "1.0.2"` |

> **Note:** `Formula/devforge.rb` is also auto-regenerated by CI with the correct sha256.  
> Update it anyway to keep the local repo in sync.

## Step-by-Step Release Process

### Step 1 — Determine the new version

Read the current version:

```bash
cat VERSION
```

Apply SemVer rules from the table above. Decide the new version (e.g., `1.0.2`).

---

### Step 2 — Update all version references

```bash
NEW_VERSION="1.0.2"   # <-- set this

# 1. VERSION file
echo "$NEW_VERSION" > VERSION

# 2. README.md badge (use Edit tool, not sed)
# Change: version-1.0.1-blue.svg  →  version-1.0.2-blue.svg

# 3. cmd/devforge-mcp/main.go
# Change: mcpserver.NewMCPServer("devforge", "1.0.1", ...)
#     to: mcpserver.NewMCPServer("devforge", "1.0.2", ...)

# 4. Formula/devforge.rb
# Change: version "1.0.1"
#     to: version "1.0.2"
```

Use the `Edit` tool for each file — do NOT use bash `sed` for Go/Ruby files.

---

### Step 3 — Verify the build compiles clean

```bash
CGO_ENABLED=1 go build ./...
```

Fix any errors before continuing.

---

### Step 4 — Write the release notes

Create a variable `RELEASE_NOTES` with valid English Markdown. Use this template:

```markdown
## What's Changed

### Bug Fixes
- Fix description of what was fixed (#issue or short description)

### Improvements
- Description of improvement

### Refactors
- Description of refactor

### Dependency Updates
- Updated X from vA to vB

---

**Full Changelog**: https://github.com/GustavoGutierrez/devforge-mcp/compare/vPREV...vNEW
```

Sections to include only if there are relevant changes:
- `### Breaking Changes` (MAJOR only — list what changed and migration path)
- `### New Features` (MINOR only — list new tools/commands)
- `### Bug Fixes`
- `### Improvements`
- `### Refactors`
- `### Dependency Updates`

---

### Step 5 — Commit and push to main

```bash
git add VERSION README.md cmd/devforge-mcp/main.go Formula/devforge.rb
git commit -m "chore(release): bump version to vX.Y.Z

<one-line summary of what changed in this release>"
git push origin main
```

Commit message format:
- Prefix: `chore(release):`
- Title: `bump version to vX.Y.Z`
- Body: one-line summary of the most important change

---

### Step 6 — Trigger the GitHub Actions release workflow

```bash
gh workflow run homebrew.yml \
  -f version=vX.Y.Z \
  --repo GustavoGutierrez/devforge-mcp
```

Replace `vX.Y.Z` with the actual version tag (must include the `v` prefix).

---

### Step 7 — Monitor the workflow

```bash
# Wait ~5s then list runs
sleep 5 && gh run list --repo GustavoGutierrez/devforge-mcp --limit 3

# Get the run ID from the output, then watch it
gh run watch <RUN_ID> --repo GustavoGutierrez/devforge-mcp
```

The workflow has three jobs:

| Job | Runner | What it does |
|-----|--------|-------------|
| `prepare-release` | ubuntu-latest | Creates GitHub Release at `vX.Y.Z` |
| `build-linux` | ubuntu-22.04 | Builds binaries, packages `.tar.gz`, uploads to release |
| `update-formula` | ubuntu-latest | Rewrites `Formula/devforge.rb`, pushes branch `update-vX.Y.Z`, opens PR |

All three must show ✓ green. If any fails, see **Troubleshooting** below.

---

### Step 8 — Verify the release and PR

```bash
# Check the GitHub Release was created
gh release view vX.Y.Z --repo GustavoGutierrez/devforge-mcp

# Check the Homebrew formula PR was opened
gh pr list --repo GustavoGutierrez/devforge-mcp --state open
```

Expected outputs:
- Release exists with a `devforge-X.Y.Z.linux-amd64.tar.gz` asset
- PR `chore(homebrew): update bottles for vX.Y.Z` is open targeting branch `homebrew-tap`

## Troubleshooting

### `unknown flag: --body` on `gh release create`
The correct flag is `--notes`, not `--body`. Check `.github/workflows/homebrew.yml` line with `gh release create`.

### `No commits between homebrew-tap and homebrew-tap`
The PR branch `update-vX.Y.Z` is identical to `homebrew-tap`. Usually means formula didn't actually change (sha256 same). Check if sha256 of the asset changed.

### `Resource not accessible by integration` on `gh pr create`
Missing `pull-requests: write` in workflow permissions **OR** the repo setting is read-only.  
Fix: **Settings → Actions → General → Workflow permissions → Allow GitHub Actions to create and approve pull requests**.

### `Process '/usr/bin/tar' failed with exit code 2`
This is a warning from the Go module cache restore step — it does NOT fail the build. Safe to ignore.

### Workflow triggered but wrong version
Confirm you passed `version=vX.Y.Z` (with `v` prefix) to `workflow_dispatch`.

### Formula version out of sync after release
The CI regenerates `Formula/devforge.rb` automatically. Run `git pull origin main` after the PR is merged to sync locally.

## CI Pipeline Architecture

```
workflow_dispatch (version=vX.Y.Z)
        │
        ▼
prepare-release ──── Creates GitHub Release vX.Y.Z
        │
        ▼
build-linux ──────── CGO_ENABLED=1 build on ubuntu-22.04 (glibc 2.35)
        │            Packages devforge-mcp + devforge + dpf → .tar.gz
        │            Uploads asset to GitHub Release
        │            Outputs sha256
        ▼
update-formula ───── Checks out homebrew-tap branch
                     Rewrites Formula/devforge.rb with new sha256
                     Pushes branch update-vX.Y.Z --force
                     Opens PR: update-vX.Y.Z → homebrew-tap
```

> macOS bottles (arm64, intel) are currently commented out in the workflow.  
> Only Linux amd64 is built. To enable macOS, uncomment the `build-arm64` and `build-intel` jobs.
